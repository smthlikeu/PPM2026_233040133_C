JAWABAN PERTANYAAN REFLEKSI - MODUL PERTEMUAN 3

1. Apa beda StatelessWidget dan StatefulWidget? Sebutkan satu contoh konkret kapan masing-masing dipakai.

Perbedaan utama antara StatelessWidget dan StatefulWidget terletak pada ada atau tidaknya state internal yang dapat berubah. StatelessWidget tidak memiliki state internal, sehingga hanya di-rebuild ketika parent widget-nya di-rebuild. StatelessWidget hanya memiliki satu method utama yaitu build(), dan lebih ringan dalam hal performa. Sebaliknya, StatefulWidget memiliki state yang dapat berubah selama widget hidup, dan dapat di-rebuild berkali-kali melalui pemanggilan setState(). StatefulWidget memiliki lifecycle yang lebih lengkap dengan method initState(), build(), dan dispose(). StatefulWidget bersifat mutable, artinya data di dalamnya dapat diubah, sedangkan StatelessWidget bersifat immutable dengan semua field bersifat final.

Contoh konkret penggunaan StatelessWidget adalah DetailCatatanPage, yang menampilkan data statis dari parameter constructor. Halaman ini tidak membutuhkan perubahan data selama halaman terbuka, hanya menampilkan informasi catatan yang telah diterima. Sebaliknya, HomePage adalah contoh penggunaan StatefulWidget karena daftar catatan dapat berubah (tambah, edit, hapus, filter), dan UI perlu di-update dengan setState() setiap kali ada perubahan data. HomePage membutuhkan state internal untuk menyimpan dan mengelola data catatan yang dinamis.

2. Mengapa kita harus memanggil setState() saat mengubah data? Apa yang terjadi kalau lupa?

setState() harus dipanggil ketika mengubah data karena method ini memberi tahu Flutter bahwa ada perubahan data di dalam state widget. Ketika setState() dipanggil, Flutter akan memicu proses rebuild pada widget tersebut sehingga UI akan diperbarui sesuai dengan nilai data terbaru. Tanpa memanggil setState(), meskipun data internal widget telah berubah, perubahan tersebut tidak akan tercermin di UI karena Flutter tidak tahu bahwa data telah berubah.

Jika lupa memanggil setState(), data memang akan berubah di memory, tetapi UI tidak akan refresh dan tetap menampilkan tampilan lama. Misalnya, jika kita menambahkan catatan baru ke list tanpa setState(), list di memory memang bertambah tetapi ListView di UI masih menampilkan jumlah item yang lama. Pengguna akan bingung karena data sudah tersimpan tetapi tidak terlihat di layar. Dalam implementasi di kode, ketika menghapus catatan, kita wrap operasi removeWhere() dengan setState(() { _semuaCatatan.removeWhere((c) => c.id == id); }) agar list yang ditampilkan langsung ter-update.

3. Apa fungsi GlobalKey<FormState>? Mengapa tidak cukup TextEditingController saja?

GlobalKey<FormState> berfungsi sebagai "remote control" untuk Form widget, memungkinkan kita mengakses dan mengontrol Form dari luar widget tersebut. Dengan GlobalKey<FormState>, kita dapat memanggil method penting seperti validate() untuk memicu validasi semua field di dalam form, save() untuk menyimpan nilai field secara terprogram, dan reset() untuk mengembalikan semua field ke nilai awal. GlobalKey juga memberikan akses ke currentState yang merupakan FormState object.

TextEditingController saja tidak cukup karena controller hanya berfungsi untuk membaca dan menulis nilai dari TextField secara terprogram. Dengan controller, kita dapat mengakses .text untuk membaca apa yang user ketikkan, tetapi kita tidak dapat memicu validasi otomatis pada field tersebut. Jika hanya menggunakan controller, kita harus melakukan validasi secara manual dengan if statement terpisah untuk setiap field, yang akan membuat kode menjadi kompleks dan sulit dikelola. GlobalKey<FormState> memungkinkan kita memanggil if (!_formKey.currentState!.validate()) return; dengan satu baris, dan Flutter akan otomatis menjalankan semua validator function di setiap TextFormField dalam form, kemudian menampilkan error message di bawah field yang gagal validasi.

4. Mengapa TextEditingController perlu di-dispose()?

TextEditingController adalah object yang mengalokasikan resource system seperti listener, memory allocation, dan stream controller di background. Ketika TextEditingController dibuat, Flutter membuat berbagai resource untuk mengelola input text dan notifikasi perubahan. Jika TextEditingController tidak di-dispose(), resource-resource tersebut akan terus menempati memory bahkan setelah widget sudah tidak lagi digunakan. Hal ini dapat menyebabkan memory leak, terutama jika form dibuka dan ditutup berkali-kali.

Proses dispose() akan membebaskan semua resource yang dialokasikan oleh TextEditingController. Analogi sederhananya adalah seperti membuka file di disk - setelah selesai menggunakan file, kita harus menutupnya agar resource file tidak terus terbuka. Demikian juga dengan database connection, setelah selesai melakukan query, connection harus di-close. Dalam implementasi, setiap TextEditingController yang dibuat di dalam State harus di-dispose di method dispose() widget. Dalam kode TambahCatatanPage, kita memiliki tiga controller: _judulCtrl, _isiCtrl, dan _emailCtrl, dan semuanya harus di-dispose di @override void dispose() dengan memanggil controller.dispose() untuk masing-masing sebelum memanggil super.dispose().

5. Apa beda Navigator.push dengan Navigator.pushNamed? Kapan pakai yang mana?

Navigator.push adalah cara navigasi yang inline dan langsung, dimana kita membuat MaterialPageRoute atau route lainnya secara inline di tempat kita memanggil navigasi. Syntax-nya adalah Navigator.push(context, MaterialPageRoute(builder: (_) => const DetailPage())). Method ini tidak memerlukan setup di awal di MaterialApp, kita bisa langsung menggunakannya di mana saja. Namun, ketika aplikasi memiliki banyak route, syntax ini menjadi berulang-ulang dan sulit untuk dilacak alur navigasinya.

Navigator.pushNamed adalah cara navigasi yang lebih terstruktur dan scalable, dimana kita mendaftarkan semua route di awal dalam MaterialApp menggunakan routes property atau onGenerateRoute callback. Syntax-nya lebih sederhana: Navigator.pushNamed(context, '/detail'). Approach ini membuat kode lebih rapi, mirip seperti URL routing di web application. Ketika aplikasi besar dengan puluhan route, menggunakan pushNamed akan membuat maintenance lebih mudah karena semua route terdefinisi di satu tempat.

Kapan menggunakan push: gunakan untuk aplikasi kecil atau prototyping cepat, atau ketika route sangat sederhana dan tidak ada dependency antar route. Kapan menggunakan pushNamed: gunakan untuk aplikasi medium hingga besar dengan multiple route, atau ketika ingin struktur code yang rapi dan scalable. Dalam implementasi tugas pertemuan 3, kita menggunakan pushNamed dengan mendaftarkan route di onGenerateRoute: switch (settings.name) { case '/tambah': return MaterialPageRoute(...); case '/detail': ... }, kemudian memanggil Navigator.pushNamed(context, '/detail', arguments: catatan).

6. Bagaimana cara mengirim data balik dari halaman B ke halaman A?

Teknik untuk mengirim data balik adalah menggunakan Navigator.pop() dengan value sebagai parameter kedua. Ketika user selesai di halaman B dan ingin kembali ke halaman A dengan membawa data, halaman B memanggil Navigator.pop(context, dataBaru). Data yang dikirim melalui pop() ini akan diterima kembali di halaman A sebagai return value dari navigasi.

Di halaman A, ketika kita navigasi ke halaman B menggunakan Navigator.pushNamed(), kita harus menggunakan await agar kode menunggu sampai halaman B memanggil pop() dan mengembalikan data. Contohnya: final hasil = await Navigator.pushNamed(context, '/tambah'); kemudian kita check if (hasil is Catatan) { setState(() => _semuaCatatan.add(hasil)); }. Alur lengkapnya adalah: HomePage navigasi ke TambahCatatanPage dengan pushNamed, kemudian menunggu dengan await. User mengisi form dan tap Simpan, form melakukan validasi. Jika valid, TambahCatatanPage membuat object Catatan baru dan memanggil Navigator.pop(context, catatanBaru). Data tersebut diterima di HomePage sebagai variable hasil, kemudian homepage melakukan setState() untuk menambahkan data ke list dan refresh UI. Setelah await selesai, kita juga perlu check if (!mounted) return; sebelum menggunakan context lagi, untuk memastikan widget masih valid dan tidak sudah di-dispose.

7. Mengapa kita perlu cek if (!mounted) return; setelah await?

Setelah kita memanggil await Navigator.pushNamed(), waktu bisa jadi lumayan lama karena user sedang berinteraksi dengan halaman B. Dalam waktu tersebut, ada kemungkinan user melakukan navigasi lain atau menutup halaman A sama sekali, sehingga halaman A sudah di-dispose dari widget tree. Jika hal ini terjadi dan kita tetap mencoba menggunakan context setelah await selesai tanpa cek, maka context yang kita gunakan sudah tidak valid dan akan menghasilkan error seperti PlatformException: MissingPluginException.

Property mounted adalah boolean flag yang otomatis diatur oleh Flutter. Ketika widget masih aktif di widget tree, mounted bernilai true, artinya context masih valid dan aman digunakan. Ketika widget sudah di-dispose atau dihapus dari widget tree, mounted akan berubah menjadi false. Oleh karena itu, setelah await selesai, kita harus selalu check if (!mounted) return; sebelum menggunakan context. Jika mounted adalah false, kita langsung return dari method dan tidak melanjutkan kode yang menggunakan context. Dalam implementasi, setelah await Navigator.pushNamed(), sebelum memanggil ScaffoldMessenger.of(context).showSnackBar(), kita harus check if (!mounted) return; terlebih dahulu. Ini adalah best practice untuk mencegah error dan memory leak di aplikasi yang robust.

RINGKASAN

Semua 7 pertanyaan refleksi telah dijawab dengan menjelaskan konsep-konsep penting dalam modul pertemuan 3. StatelessWidget digunakan untuk tampilan statis tanpa perubahan data, sementara StatefulWidget digunakan untuk data yang dinamis dan perlu ui update dengan setState(). GlobalKey<FormState> adalah kontrol utama untuk form yang berbeda dengan TextEditingController yang hanya berfungsi membaca/menulis nilai. TextEditingController harus di-dispose untuk mencegah memory leak. Navigator.push cocok untuk app kecil sementara Navigator.pushNamed lebih baik untuk app besar. Data bisa dikirim balik dari halaman B ke A menggunakan Navigator.pop() dengan await. Setelah await, selalu check mounted sebelum menggunakan context untuk menghindari error. Semua konsep ini sudah diimplementasikan dalam project Tugas_P4 beserta 3 tugas mandiri: fitur edit catatan, filter berdasarkan kategori, dan validasi email dengan regex.
